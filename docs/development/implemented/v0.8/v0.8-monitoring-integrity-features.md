# v0.8 — Monitoring & Assessment Integrity Features

**Status:** Implemented  
**Date:** 2026-05-03

## Overview

This phase introduces a suite of monitoring and integrity controls designed to detect technical issues during assessments and deter/record gaming behaviour. It spans the full stack: database schema, Python voice engine domain layer, FastAPI routes, Next.js API proxies, and frontend components.

---

## Features Implemented

### 1. Browser Focus Detection

**What it does:** Records every time a candidate's browser window loses focus during a live call, including the assessment phase at the time and the duration of each absence.

**Flagging logic:** A session is marked `focus_suspicious = true` when either:
- 5 or more focus-loss events occur, **or**
- Any single absence exceeds 60 seconds

**Why it works this way:** Individual brief absences (notification glance, alt-tab) are not meaningful signals. Repeated losses or extended absences during evidence gathering are. The threshold avoids noise while catching sustained off-window behaviour.

**Data stored:** `focus_events_json` (JSONB array of `{at, phase, durationMs}`), `focus_suspicious` (boolean), `total_focus_away_ms` (integer).

**Frontend:** `visibilitychange` listener in `CallStateDisplay` fires only while `status === "in_progress"`. Events are sent via `POST /api/assessment/{sessionId}/focus-event`.

---

### 2. Deterrent Messaging

**What it does:** Displays a notice below the intake form submit button making explicit that:
- Focus changes are monitored during the call
- All claims made will be verified by management and a qualified SME

**Design decision:** No checkbox — the message is informational and determinrent-focused. The existing consent paragraph (recording and analysis) remains separate. The bot's verbal introduction script should also include this notice when Phase 4 SFIAFlowController is implemented.

---

### 3. Progressive Transcript Saving

**New endpoint:** `POST /api/v1/assessment/{session_id}/transcript-turn`

**What it does:** Appends one transcript turn to `transcript_json.turns` and updates `last_turn_saved_at` after every bot/candidate utterance. Previously, the full transcript was written only at call end — a crash mid-call would lose everything.

**Admin visibility:** `last_turn_saved_at` lets an admin see whether an in-progress session is actively advancing or has silently stalled.

**Future:** The saved phase state enables a resume path for Phase 4 once the SFIAFlowController is stable.

---

### 4. Structured Termination Logging

**New column:** `termination_reason VARCHAR(50)` + `error_details JSONB` on `assessment_sessions`.

**Values:**

| Reason | Status set to | Counts toward cooldown |
|--------|--------------|----------------------|
| `user_ended` | `user_ended` | Yes |
| `browser_closed` | `user_ended` | Yes |
| `timeout_no_audio` | `failed` | No |
| `timeout_duration` | `failed` | No |
| `websocket_error` | `failed` | No |
| `llm_error` | `failed` | No |
| `transport_error` | `failed` | No |
| `admin_cancelled` | `cancelled` | No |

**Status distinction:** The `user_ended` status value was added to differentiate "candidate intentionally ended the call" from `cancelled` (pre-call consent declined, pre-dial abort) and `failed` (technical error). This matters for the cooldown gate.

**Frontend:** Cancel button sends `{ termination_reason: "user_ended" }`. The `beforeunload` listener fires a `navigator.sendBeacon` with `browser_closed` when the tab is closed mid-call.

**Error details:** On `_place_call` failure and `on_call_failed`, the exception type and message are written to `error_details` for debugging without needing log correlation.

### 5. Assessment Cooldown / Repeat Control

#### Database
- `Candidate` gains: `cooldown_override_granted_at`, `cooldown_override_expires_at`, `no_restrictions`
- New `AdminSettings` singleton table: `cooldown_days` (default 90)
- New `CandidateRestrictionAudit` table: immutable log of every admin action

#### Eligibility service (`AssessmentEligibilityService`)
- Called in `CallManager.trigger_call` — raises `EligibilityError` (HTTP 429) when ineligible
- Returns: `eligible`, `reason`, `next_eligible_at`, `cooldown_days`
- Technical failures (`failed` status) are **never** counted toward the cooldown
- Eligible bypass paths (checked in order):
  1. `no_restrictions = true` — always eligible
  2. `cooldown_override_expires_at > now` — one-time bypass window active
  3. No `completed`/`processed`/`user_ended` session within the cooldown window

#### Admin controls

| Endpoint | Effect |
|----------|--------|
| `GET /api/v1/admin/settings` | Read cooldown_days |
| `PUT /api/v1/admin/settings` | Update cooldown_days (1–3650 days) |
| `GET /api/v1/admin/candidates/{id}/restrictions` | Read state + audit log |
| `POST /api/v1/admin/candidates/{id}/override` | Grant time-limited bypass (default 7 days) |
| `DELETE /api/v1/admin/candidates/{id}/override` | Revoke active override |
| `PATCH /api/v1/admin/candidates/{id}/no-restrictions` | Toggle unrestricted flag |
| `GET /api/v1/assessment/eligibility?candidate_id=` | Pre-flight eligibility check |

**Design notes:**
- The one-time override is time-scoped (not permanent) — defaults to 7 days, max 365. This prevents forgotten grants.
- Every grant/revoke/toggle writes an audit row so disputes can be investigated.
- `no_restrictions` is a named flag (not just a zero cooldown) so its intent is explicit in the UI and audit log.
- The frontend should call `/api/v1/assessment/eligibility` before showing the trigger form and display the `next_eligible_at` date if ineligible.
- Partially completed calls (pre-call `cancelled`) do not count. Only calls where the candidate actually connected (status `user_ended`, `completed`, `processed`) count.

## Schema Changes

```sql
-- assessment_sessions (new columns)
ALTER TABLE assessment_sessions
  ADD COLUMN termination_reason  VARCHAR(50),
  ADD COLUMN error_details       JSONB,
  ADD COLUMN last_turn_saved_at  TIMESTAMPTZ,
  ADD COLUMN focus_suspicious    BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN total_focus_away_ms INT NOT NULL DEFAULT 0,
  ADD COLUMN focus_events_json   JSONB;

-- candidates (new columns)
ALTER TABLE candidates
  ADD COLUMN cooldown_override_granted_at TIMESTAMPTZ,
  ADD COLUMN cooldown_override_expires_at TIMESTAMPTZ,
  ADD COLUMN no_restrictions              BOOLEAN NOT NULL DEFAULT FALSE;

-- new tables
CREATE TABLE admin_settings (
  id            TEXT PRIMARY KEY DEFAULT 'default',
  "cooldownDays" INT NOT NULL DEFAULT 90,
  "updatedAt"   TIMESTAMPTZ NOT NULL,
  "updatedBy"   VARCHAR(255)
);

CREATE TABLE candidate_restriction_audit (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  "candidateId" VARCHAR(255) NOT NULL REFERENCES candidates(email) ON DELETE CASCADE,
  action       VARCHAR(50) NOT NULL,
  "grantedBy"  VARCHAR(255) NOT NULL,
  "expiresAt"  TIMESTAMPTZ,
  reason       TEXT,
  "createdAt"  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## Architecture Compliance

- **ADR-001 (Hexagonal):** `AssessmentEligibilityService` is a pure domain service — no external imports, injected `IPersistence` via constructor.
- **ADR-001 (Hexagonal):** All new persistence methods defined as `@abstractmethod` on `IPersistence` port; both `InMemoryPersistence` and `PostgresPersistence` implement them.
- **ADR-002 (Prisma @@map):** Both new models (`AdminSettings`, `CandidateRestrictionAudit`) declare `@@map` directives.
- **Domain isolation:** `eligibility_service.py` imports only from `src.domain.*` — passes Check 6.
